# WARP.md

This file provides guidance to WARP (warp.dev) when working with code in this repository.

## Common commands

### Setup
```bash
cp .env.example .env
# Edit .env (DB_*, JWT_*, etc.)

go mod download
go mod tidy
```

### Run the API server (local)
```bash
go run .
# or: go run main.go
```

### Build
```bash
mkdir -p bin
go build -o bin/kynettic-backend .
./bin/kynettic-backend
```

### Format
```bash
go fmt ./...
# If you need to rewrite files directly:
gofmt -w .
```

### Lint / static checks
This repo does not currently include a `.golangci.*` config; `golangci-lint` runs with defaults.
```bash
# If installed:
golangci-lint run ./...

# Built-in checks:
go vet ./...
```

### Tests
There are currently no `*_test.go` files in this repo; when tests are added:
```bash
go test ./...

# Run a single package:
go test ./internal/service

# Run a single test by name (example):
go test ./internal/service -run TestName
```

### Swagger / OpenAPI
Swagger is served at `GET /swagger/*any` via `gin-swagger`, with generated docs under `docs/`.

Regenerate docs after changing handler annotations:
```bash
# Install generator if needed:
go install github.com/swaggo/swag/cmd/swag@latest

swag init -g main.go -o docs
```

If Swagger UI/registration is broken, there is a small debug helper:
```bash
go run ./cmd/debug_swagger
```

### Redis connectivity smoke test
There is a small script to validate Redis operations:
```bash
go run ./scripts/test_redis.go
```

## High-level architecture

### Composition root
`main.go` is the composition root. Startup flow:
1. Load configuration via `internal/config` (Viper; reads `.env` if present, otherwise uses environment).
2. Initialize logging via `pkg/logger` (Zap).
3. Initialize request validation helpers via `internal/utils`.
4. Initialize database via `internal/database` (GORM + postgres) and run `AutoMigrate()` when `DB_AUTO_MIGRATE=true`.
5. Initialize Redis via `internal/cache` when `REDIS_ENABLED=true`.
6. Initialize external API clients under `internal/clients/*` (currently placeholder clients).
7. Wire repositories (`internal/repository`) → services (`internal/service`) → handlers (`internal/handler`).
8. Create the Gin engine via `internal/router` and start an `http.Server` with graceful shutdown.

### HTTP request flow (Gin)
Routing is centralized in `internal/router/router.go`:
- Global middleware: `Recovery()`, `Logger()`, `CORS(cfg)`, `RateLimit(cfg)`.
- Unversioned endpoints:
  - `GET /health` (checks DB + Redis)
  - `GET /swagger/*any` (Swagger UI)
- Versioned API group: `/api/v1`
  - Public: `/auth/*` (register/login/refresh)
  - Protected: `/users/*` guarded by `middleware.Auth(cfg)`
  - Admin-only: `/users` list/get/delete guarded by `middleware.RequireRole(models.RoleAdmin)`

Auth middleware parses `Authorization: Bearer <token>`, validates JWT via `internal/utils`, and stores claims in Gin context under key `"user"` (`middleware.UserContextKey`).

### Layering pattern
This repo follows a fairly strict layering:
- `internal/handler`: HTTP boundary (Gin), request binding/validation, response formatting.
- `internal/service`: business logic (e.g. auth, user management).
- `internal/repository`: persistence primitives (GORM queries).
- `internal/models`: database models and GORM hooks (e.g. hashing passwords in `User.BeforeCreate/BeforeUpdate`).
- `internal/utils`: shared helpers (JWT, password hashing, response types, validation formatting).

### Persistence & migrations
- Postgres is configured via `internal/config.Config.GetDSN()`.
- GORM is initialized in `internal/database.Initialize(cfg)`.
- `internal/database.AutoMigrate()` enumerates all models to be migrated; changing the schema usually means updating models and (if needed) adding to this migrate list.

### Cache
`internal/cache` wraps `redis/go-redis` with convenience methods (`SetJSON`, `GetJSON`, etc.) and is used by `internal/handler/health_handler.go` for dependency checks.

### Generated code
`docs/` (especially `docs/docs.go`) is generated by `swag init` and should not be edited by hand.

## Configuration notes (what must be set for the server to start)
Configuration is validated in `internal/config.(*Config).Validate()`.

Required values:
- `DB_HOST`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` (and optionally `DB_PORT`, `DB_SSLMODE`)
- `JWT_SECRET`, `JWT_REFRESH_SECRET` (and optionally expirations)

Redis:
- Controlled by `REDIS_ENABLED` (default is `true` in code). If enabled, `REDIS_HOST` must be set.

External clients (currently placeholders):
- Blockradar: `BLOCKRADAR_API_KEY`, `BLOCKRADAR_BASE_URL`
- Nomba: `NOMBA_CLIENT_ID`, `NOMBA_CLIENT_SECRET`, `NOMBA_BASE_URL`, `NOMBA_ACCOUNT_ID`
